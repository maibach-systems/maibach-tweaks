WORKFLOW — match ceremony to task size. Don't over-engineer simple work.

SIMPLE (obvious fix, 1-3 files, clear scope):
Just do it. Read, edit, deploy. No plan, no TaskCreate, no subagent ritual.

MEDIUM (3-5 files, known area):
Short plan in chat. Implement. Grep-verify if patterns/imports changed. Deploy.

COMPLEX (5+ files, refactor, new feature, unclear scope, or user asks for plan):
1. Plan to .claude/current-plan.md — group coupled phases into DEPLOY GROUPS. Each group ends with "-> DEPLOY + STOP". Phases within a group are worked through without stopping. Single independent phases = own group.
2. plan-griller validation: Spawn plan-griller subagent (subagent_type: plan-griller, model: opus — exception to the sonnet rule). Wait for results. Fix problems found. This is NOT optional — the hook enforces it.
3. Present plan + validation results to user. Wait for confirm.
4. TaskCreate the deploy groups. Update status as you go.
5. Within a group: work through all phases, grep-verify each. At "-> DEPLOY + STOP": run check-changes subagent (model: sonnet), fix issues, then deploy following the project's deploy rules (see project CLAUDE.md), then STOP and wait for "weiter".

IMPORTANT: "check-changes" means the /check-changes skill (Build + Regressions + Reuse + Plan-Check). Do NOT substitute code-quality or ops-reviewer — those only check readability, not correctness.

SUBAGENTS — use aggressively when they actually help:
- Explore agent for codebase research (instead of inline greps across many files)
- Parallel code-review + security review on complex changes
- Grep-verify in separate agent (avoids confirmation bias)
- Not mandatory for simple work — judgment call based on task complexity.

PARALLEL TOOLS — always batch independent calls:
- Multiple file reads in one message
- Git status + diff + log together
- Independent edits across files in parallel

ALWAYS: CLAUDE.md + .claude/rules/ are binding when working in a project.
